1. Programming Languages & Core Tools

1.1 C/C++ — The Performance Layer

Why: Flight controllers run on microcontrollers with limited RAM and CPU. Every microsecond counts. Real-time constraints demand predictable, fast code. C/C++ powers PX4, ArduPilot, and onboard controllers.

C/C++ is the backbone of flight-critical code. Your drone's attitude controller, sensor fusion, and motor mixing must execute in tight loops at 400+ Hz. Python's garbage collector can cause 100ms pauses—fatal for a quadcopter doing 20 m/s.

Key Concepts for Drone Flight Control

  • Pointers & memory management: manual RAM allocation for real-time safety
  • Interrupt handlers: respond to sensor events immediately
  • Fixed-point math: faster than floating-point on some microcontrollers
  • DMA (Direct Memory Access): offload data transfers to free the CPU

Example: Simple PID Controller in C

C — Flight Controller
typedef struct {
    float kp, ki, kd;
    float integral, prev_error;
} PIDController;

float pid_update(PIDController *pid, float setpoint, float measured, float dt) {
    float error = setpoint - measured;
    pid->integral += error * dt;
    float derivative = (error - pid->prev_error) / dt;

    float output = pid->kp * error
                 + pid->ki * pid->integral
                 + pid->kd * derivative;

    pid->prev_error = error;
    return output;
}

// Usage in main loop (400 Hz)
void attitude_control_loop(void) {
    IMU_read(&gyro, &accel);
    float roll_rate_cmd = pid_update(&roll_pid, roll_setpoint, roll_current, 0.0025); // 400 Hz
    mixer_update(roll_rate_cmd, pitch_rate_cmd, yaw_rate_cmd, thrust_cmd);
}

Example: Reynolds' Boids (Swarm Behavior Simulation)

Python — Swarm Dynamics
import numpy as np
import matplotlib.pyplot as plt

class Boid:
    def __init__(self, x, y, vx, vy):
        self.pos = np.array([x, y])
        self.vel = np.array([vx, vy])
        self.acc = np.array([0.0, 0.0])

    def apply_rules(self, boids, separation_d=15, alignment_d=50, cohesion_d=100):
        # Rule 1: Separation (avoid crowding)
        steer = np.array([0.0, 0.0])
        count = 0
        for other in boids:
            dist = np.linalg.norm(self.pos - other.pos)
            if 0 < dist < separation_d:
                steer -= (other.pos - self.pos)
                count += 1
        if count > 0:
            steer /= count
            self.acc += steer * 0.05

        # Rule 2: Alignment (steer toward avg heading)
        avg_vel = np.mean([b.vel for b in boids if np.linalg.norm(self.pos - b.pos) < alignment_d], axis=0)
        if avg_vel.any():
            self.acc += (avg_vel - self.vel) * 0.02

        # Rule 3: Cohesion (steer toward center of mass)
        nearby = [b.pos for b in boids if np.linalg.norm(self.pos - b.pos) < cohesion_d]
        if nearby:
            center = np.mean(nearby, axis=0)
            self.acc += (center - self.pos) * 0.01

    def update(self, dt=0.1):
        self.vel += self.acc
        self.vel = np.clip(self.vel, -5, 5)  # Max speed
        self.pos += self.vel * dt
        self.acc = np.array([0.0, 0.0])

# Simulation
n_boids = 50
boids = [Boid(np.random.rand()*400, np.random.rand()*400,
              (np.random.rand()-0.5)*2, (np.random.rand()-0.5)*2)
         for _ in range(n_boids)]

for t in range(1000):
    for boid in boids:
        boid.apply_rules(boids)
    for boid in boids:
        boid.update()

1.3 ROS / ROS2 (Robot Operating System)

Why: ROS is the standard middleware for multi-robot systems. It decouples your flight controller from ground station, sensor fusion from path planning. Perfect for coordinating 10 or 100 drones without spaghetti code.

ROS is an operating system for robots. It handles inter-process communication, hardware abstraction, and message passing. Think of it as middleware that lets drone #1's position feed into drone #2's collision avoidance without them being in the same program.

Core ROS Concepts for Swarms

  • Topics: publish-subscribe messaging (like a radio broadcast)
  • Services: request-response communication (like a phone call)
  • Actions: long-running tasks with feedback (like ordering delivery)
  • tf2: coordinate frame management (drone frame, world frame, camera frame)
  • Launch files: orchestrating 50+ nodes with one command

2. Flight Controller Firmware

2.1 PX4 Autopilot

Why: PX4 is the most widely used open-source flight controller. It's modular, well-documented, and has built-in multi-vehicle support. Its offboard mode lets external computers send commands in a real-time-safe way.

PX4 runs on most drone flight controllers (Pixhawk, Cube, etc.). It's not a monolithic program but a set of interchangeable modules: sensors → estimator → controller → mixer. Each module publishes/subscribes to messages through uORB.

PX4 Architecture Overview

Sensors (IMU, GPS, Barometer)
    ↓ (uORB messages)
Estimators (EKF2, attitude filter)
    ↓
Controllers (Position, Velocity, Attitude)
    ↓
Mixer (Motor PWM output)
    ↓
Motors

Key PX4 Concepts

  • uORB: Internal publish-subscribe messaging (faster than ROS on the drone)
  • EKF2: Extended Kalman Filter for sensor fusion (magic that turns noisy sensors into reliable state estimates)
  • Offboard Mode: External computer sends velocity/attitude setpoints via MAVLink
  • Control Loops: 400 Hz rate control, 50 Hz position control
  • Motor Mixer: Converts thrust + moment commands into per-motor PWM

Example: PX4 Offboard Control via MAVROS

C++ — Offboard Position Control
#include <ros/ros.h>
#include <mavros_msgs/SetMode.h>
#include <mavros_msgs/CommandBool.h>
#include <geometry_msgs/PoseStamped.h>

int main(int argc, char** argv) {
    ros::init(argc, argv, "offboard_control");
    ros::NodeHandle nh;

    ros::ServiceClient set_mode_client = nh.serviceClient<mavros_msgs::SetMode>
        ("/mavros/set_mode");
    ros::ServiceClient arming_client = nh.serviceClient<mavros_msgs::CommandBool>
        ("/mavros/cmd/arming");
    ros::Publisher local_pos_pub = nh.advertise<geometry_msgs::PoseStamped>
        ("/mavros/setpoint_position/local", 10);

    ros::Rate rate(20.0);

    // Wait for connection
    while (ros::ok() && !current_state.connected) {
        ros::spinOnce();
        rate.sleep();
    }

    // Set to offboard mode & arm
    mavros_msgs::SetMode offboard_set_mode;
    offboard_set_mode.request.custom_mode = "OFFBOARD";
    set_mode_client.call(offboard_set_mode);

    mavros_msgs::CommandBool arm_cmd;
    arm_cmd.request.value = true;
    arming_client.call(arm_cmd);

    // Send setpoint commands
    geometry_msgs::PoseStamped pose;
    pose.pose.position.x = 0;
    pose.pose.position.y = 0;
    pose.pose.position.z = 1.0;  // Hover at 1m

    while (ros::ok()) {
        local_pos_pub.publish(pose);
        ros::spinOnce();
        rate.sleep();
    }

    return 0;
}

ArduPilot Lua Script Example

Lua — On-Board Custom Behavior
-- Custom swarm behavior on ArduPilot flight controller
SWARM_ID = 1
NEIGHBOR_MAX_DIST = 20  -- meters
UPDATE_RATE = 10  -- Hz

function update()
    -- Read current position
    local pos = gps:location()
    local velocity = ahrs:get_velocity_NED()

    -- In production, receive neighbor positions from radio
    -- Here we simulate a formation hold

    -- Read target formation position from telemetry (from ground station)
    local target_x = param:get('SWARM_X')
    local target_y = param:get('SWARM_Y')
    local target_z = param:get('SWARM_Z')

    -- Calculate error
    local err_x = target_x - pos:x()
    local err_y = target_y - pos:y()
    local err_z = target_z - pos:z()

    -- Use simple P controller
    local vel_x = err_x * 0.5
    local vel_y = err_y * 0.5
    local vel_z = err_z * 0.5

    -- Send velocity command
    vehicle:set_target_velocity_body(vel_x, vel_y, vel_z)

    return update, 1000/UPDATE_RATE  -- 10 Hz update
end

return update()

2.3 Betaflight / Cleanflight

Betaflight focuses on racing and acrobatic flight. It prioritizes low-latency rate control over stable autonomous flight. Not ideal for swarms but teaches valuable PID tuning lessons. Key concept: rate mode (gyro stabilization only) vs. angle mode (gyro + accelerometer for level hold).

2.4 Custom Flight Controllers

Why: Research. Special requirements (custom sensors, unique dynamics). When open-source firmware doesn't fit your needs.

Minimum Viable Flight Controller (Pseudocode)

Pseudocode — Flight Controller Loop
void main_loop() {
    while (true) {
        // Step 1: Read sensors (1000 Hz or higher)
        imu.read();           // 9-axis: gyro + accel + mag
        gps.read();           // 10 Hz update
        barometer.read();     // 50 Hz update

        // Step 2: Estimate state (400 Hz)
        // Complementary filter: gyro (fast, drifts) + accel (slow, noisy)
        angle_roll = 0.98 * (angle_roll + gyro_roll * dt) + 0.02 * accel_roll;
        angle_pitch = 0.98 * (angle_pitch + gyro_pitch * dt) + 0.02 * accel_pitch;

        // Step 3: Attitude controller (400 Hz)
        // Convert desired attitude → desired rate
        float roll_rate_cmd = pid_roll.update(desired_roll, angle_roll);
        float pitch_rate_cmd = pid_pitch.update(desired_pitch, angle_pitch);
        float yaw_rate_cmd = pid_yaw.update(desired_yaw, angle_yaw);

        // Step 4: Rate controller (400 Hz)
        // Convert desired rate → motor commands
        float roll_moment = pid_rate_roll.update(roll_rate_cmd, gyro_roll);
        float pitch_moment = pid_rate_pitch.update(pitch_rate_cmd, gyro_pitch);
        float yaw_moment = pid_rate_yaw.update(yaw_rate_cmd, gyro_yaw);

        // Step 5: Motor mixing (400 Hz)
        // Quadcopter X configuration:
        // motor[0] = front-left    motor[2] = front-right
        // motor[1] = back-left     motor[3] = back-right
        motor[0] = thrust - roll_moment + pitch_moment + yaw_moment;
        motor[1] = thrust - roll_moment - pitch_moment - yaw_moment;
        motor[2] = thrust + roll_moment + pitch_moment - yaw_moment;
        motor[3] = thrust + roll_moment - pitch_moment + yaw_moment;

        // Clamp to valid range [1000, 2000] microseconds
        for (int i = 0; i < 4; i++) {
            motor[i] = constrain(motor[i], 1000, 2000);
        }

        // Step 6: Output PWM
        pwm_write(motor);

        // Wait for next cycle (2.5 ms for 400 Hz)
        delay_us(2500);
    }
}

4. Communication Protocols

4.1 MAVLink

Why: The standard protocol for all drone communication. Works with PX4, ArduPilot, every ground station. Binary format, minimal overhead, extensible message definitions.

MAVLink is a lightweight serialization protocol for drone-ground communication. Each message has a header, payload, and checksum. Designed for low-bandwidth links (like 433 MHz radio or serial).

MAVLink Message Structure

Frame: STX | LEN | SEQ | SYSID | COMPID | MSGID | PAYLOAD | CKA | CKB

STX:     1 byte   = 0xFE (frame start)
LEN:     1 byte   = payload length
SEQ:     1 byte   = sequence number (detects dropped packets)
SYSID:   1 byte   = source system ID (e.g., 1=vehicle, 255=ground)
COMPID:  1 byte   = component ID (e.g., 1=autopilot)
MSGID:   1 byte   = message type ID
PAYLOAD: variable = actual data
CKA, CKB:2 bytes  = CRC checksum

Key MAVLink Messages for Swarms

Message Direction Frequency Purpose
HEARTBEAT Drone → Ground 1 Hz "I'm alive" signal
ATTITUDE Drone → Ground 50 Hz Roll, pitch, yaw, rates
GLOBAL_POSITION_INT Drone → Ground 10 Hz GPS lat/lon, altitude
SET_POSITION_TARGET_LOCAL_NED Ground → Drone 50 Hz Position setpoint (offboard mode)
SET_ATTITUDE_TARGET Ground → Drone 50 Hz Attitude + thrust setpoint
LOCAL_POSITION_NED Drone → Ground 50 Hz Position in local frame (meters)

MAVLink Message Flow Example

Ground Station                  Drone (PX4)
     |                              |
     |---COMMAND_LONG ARM---------->|
     |                              | Arm motors
     |<----------COMMAND_ACK--------|

4.3 Mesh Networking

Why: Real swarms can't rely on a ground station within range. Drones must relay data through each other to form a mesh network.

Inter-Drone Communication Options

Tech Range Bandwidth Power Swarm Use
WiFi Ad-Hoc 100m 54 Mbps High Indoor swarms, high-bandwidth
XBee 1-3 km 250 kbps Low Long-distance coordination
LoRa 10+ km 50 kbps Low Ultra-long-range swarms (rare)
5G/LTE Cellular 100+ Mbps High Urban swarms with infrastructure
433 MHz Radio 1-10 km 10-100 kbps Low Cheap, reliable, low-bandwidth

Swarm Communication Design Rules

  • What to share: position, velocity, planned trajectory (as polynomial coefficients), sensor data, task status
  • Update rates: 10-50 Hz for position, 1-5 Hz for high-level commands, 0.1 Hz for task updates
  • Compression: Send polynomial coefficients of trajectory instead of full waypoint list (100 bytes → 20 bytes)
  • Neighbor discovery: Each drone broadcasts "I exist" beacons; listen for neighbors within range
  • Graceful degradation: If communication drops, drones continue with last known plan for 5 seconds

PyTorch Swarm RL Example (Flocking Policy)

Python — PyTorch Multi-Agent RL
import torch
import torch.nn as nn
from pettingzoo.butterfly import pistonball_v6

# Create environment
env = pistonball_v6.env(n_pistons=5)

# Define policy network (shared across agents)
class PolicyNetwork(nn.Module):
    def __init__(self, obs_dim, action_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(obs_dim, 128),
            nn.ReLU(),
            nn.Linear(128, 128),
            nn.ReLU(),
            nn.Linear(128, action_dim)
        )

    def forward(self, obs):
        return self.net(obs)

obs_dim = 24  # From environment
action_dim = env.action_spaces['piston_0'].n  # Discrete actions

policy = PolicyNetwork(obs_dim, action_dim)
optimizer = torch.optim.Adam(policy.parameters(), lr=1e-3)

# Training loop
for episode in range(1000):
    observations, infos = env.reset()
    episode_reward = 0

    for step in range(100):
        actions = {}
        log_probs = []

        # Get action for each agent
        for agent in env.agents:
            obs = torch.FloatTensor(observations[agent])
            logits = policy(obs)
            action_probs = torch.softmax(logits, dim=-1)
            action = torch.multinomial(action_probs, 1).item()
            actions[agent] = action
            log_probs.append(torch.log(action_probs[action]))

        # Step environment
        observations, rewards, terminations, truncations, infos = env.step(actions)

        # Simple policy gradient update
        total_reward = sum(rewards.values())
        episode_reward += total_reward

        loss = -torch.stack(log_probs).sum() * total_reward
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

    print(f"Episode {episode}: {episode_reward:.2f}")

env.close()

from stable_baselines3 import PPO
from gymnasium import Env
import numpy as np

# Define custom swarm environment
class SwarmEnv(Env):
    def __init__(self, num_drones=5):
        super().__init__()
        self.num_drones = num_drones

        # Observation: position & velocity of all drones
        self.observation_space = gymnasium.spaces.Box(
            low=-100, high=100,
            shape=(num_drones * 6,),  # x, y, z, vx, vy, vz
            dtype=np.float32
        )

        # Action: velocity command for each drone
        self.action_space = gymnasium.spaces.Box(
            low=-1, high=1,
            shape=(num_drones * 3,),  # vx, vy, vz per drone
            dtype=np.float32
        )

    def reset(self):
        # Random initial positions
        self.state = np.random.randn(self.num_drones * 6) * 10
        return self.state, {}

    def step(self, action):
        # Update positions based on velocity commands
        for i in range(self.num_drones):
            self.state[i*6:i*6+3] += action[i*3:i*3+3] * 0.1

        # Reward: minimize distance between drones (formation)
        positions = self.state[:self.num_drones*3].reshape(self.num_drones, 3)
        distances = np.linalg.norm(positions[:, None, :] - positions[None, :, :], axis=2)
        separation_reward = -np.mean(distances[distances > 0])  # Penalize large gaps

        # Penalty for high velocity (smooth motion)
        velocity_penalty = -0.01 * np.sum(action**2)

        reward = separation_reward + velocity_penalty
        terminated = False
        truncated = False
        info = {}

        return self.state, reward, terminated, truncated, info

# Train agent
env = SwarmEnv(num_drones=5)
model = PPO("MlpPolicy", env, verbose=1)
model.learn(total_timesteps=100000)

# Save model
model.save("swarm_formation_ppo")

# Test trained model

    action, _states = model.predict(obs, deterministic=True)

6.4 Gymnasium (OpenAI Gym)

Standard interface for RL environments. Define observation_space, action_space, reset(), step(). Compatible with all RL libraries (Stable Baselines, TF-Agents, RLlib).

Custom Swarm Gym Environment (Skeleton)

Python — Gymnasium Template
import gymnasium as gym
import numpy as np

class MultiDroneEnv(gym.Env):
    """
    Multi-drone formation control environment.
    Observation: [pos_x, pos_y, pos_z, vel_x, vel_y, vel_z] for each drone
    Action: [cmd_vx, cmd_vy, cmd_vz] for each drone
    """

    metadata = {"render_modes": ["human"]}

    def __init__(self, num_drones=4, max_steps=1000, render_mode=None):
        super().__init__()
        self.num_drones = num_drones
        self.max_steps = max_steps
        self.render_mode = render_mode
        self.step_count = 0

        # Observation space: [x,y,z,vx,vy,vz] * num_drones
        obs_size = num_drones * 6
        self.observation_space = gym.spaces.Box(
            low=-200, high=200, shape=(obs_size,), dtype=np.float32
        )

        # Action space: [vx_cmd,vy_cmd,vz_cmd] * num_drones
        action_size = num_drones * 3
        self.action_space = gym.spaces.Box(
            low=-5, high=5, shape=(action_size,), dtype=np.float32
        )

    def reset(self, seed=None):
        super().reset(seed=seed)
        # Initialize drone state
        self.state = np.random.randn(self.num_drones * 6) * 5
        self.step_count = 0
        return self.state.copy(), {}

    def step(self, action):
        self.step_count += 1

        # Physics simulation (simple Euler integration)
        dt = 0.1
        positions = self.state[: self.num_drones * 3].reshape(self.num_drones, 3)
        velocities = self.state[self.num_drones * 3 :].reshape(self.num_drones, 3)

        # Velocity commands from RL agent
        cmd_velocities = action.reshape(self.num_drones, 3)

        # Update velocities (1st order lag)
        velocities = 0.9 * velocities + 0.1 * cmd_velocities

        # Update positions
        positions = positions + velocities * dt

        # Update state
        self.state[: self.num_drones * 3] = positions.flatten()
        self.state[self.num_drones * 3 :] = velocities.flatten()

        # Compute reward
        reward = self._compute_reward()

        terminated = self.step_count >= self.max_steps
        truncated = False

        return self.state.copy(), reward, terminated, truncated, {}

    def _compute_reward(self):
        # Reward for maintaining formation (drones stay together)
        positions = self.state[: self.num_drones * 3].reshape(self.num_drones, 3)
        pairwise_dist = np.linalg.norm(
            positions[:, None, :] - positions[None, :, :], axis=2
        )

        # Ideal separation = 2 meters
        formation_reward = -np.mean((pairwise_dist - 2.0) ** 2)

        # Penalty for collisions (distance < 0.5m)
        collision_penalty = -100 * np.sum(pairwise_dist < 0.5)

        # Penalty for excessive control effort
        action_penalty = -0.01 * np.sum(self.action ** 2)

        return formation_reward + collision_penalty + action_penalty

        if self.render_mode == "human":
            # TODO: Implement 3D visualization
